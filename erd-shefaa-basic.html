ERD نظام Shefaa - النسخة الأساسية

هذا الملف يشرح النسخة المختصرة من نموذج البيانات الخاصة بنظام Shefaa بشكل واضح ومناسب للعرض أو التقرير.

1. شرح الكيانات الأساسية

USER

تمثل المستخدم الأساسي في النظام، سواء كان مريضًا أو طبيبًا أو مسؤولًا.

  • id: مفتاح أساسي
  • email: البريد الإلكتروني
  • firstName: الاسم الأول
  • lastName: الاسم الأخير
  • phoneNumber: رقم الهاتف
  • role: الدور داخل النظام

PATIENT

يمثل البيانات الخاصة بالمريض داخل النظام.

  • id: مفتاح أساسي
  • userId: رابط إلى المستخدم
  • medicalRecordNumber: رقم السجل الطبي

DOCTOR

يمثل البيانات الخاصة بالطبيب.

  • id: مفتاح أساسي
  • userId: رابط إلى المستخدم
  • specialtyId: التخصص الرئيسي للطبيب
  • licenseNumber: رقم الرخصة الطبية

SPECIALTY

يمثل التخصصات الطبية مثل القلب أو الجلدية أو الأطفال.

  • id: مفتاح أساسي
  • name: اسم التخصص

CLINIC

يمثل العيادة أو المركز الطبي.

  • id: مفتاح أساسي
  • name: اسم العيادة
  • address: العنوان
  • city: المدينة

CLINIC_DOCTOR

جدول وسيط يربط الطبيب بالعيادة التي يعمل بها.

  • clinicId: رقم العيادة
  • doctorId: رقم الطبيب

APPOINTMENT

يمثل الحجز الطبي بين المريض والطبيب في عيادة معينة.

  • patientId: المريض
  • doctorId: الطبيب
  • clinicId: العيادة
  • scheduledStart: وقت بداية الحجز
  • scheduledEnd: وقت نهاية الحجز
  • status: حالة الحجز مثل Pending أو Confirmed

MEDICAL_RECORD

يمثل السجل الطبي للمريض بعد أو أثناء الزيارة.

  • appointmentId: الحجز المرتبط
  • patientId: المريض
  • doctorId: الطبيب
  • diagnosis: التشخيص

3. شرح العلاقات

  • USER → PATIENT: كل مستخدم قد يكون مريضًا واحدًا.
  • USER → DOCTOR: كل مستخدم قد يكون طبيبًا واحدًا.
  • SPECIALTY → DOCTOR: كل تخصص يمكن أن يختص به عدة أطباء.
  • DOCTOR ↔ CLINIC: الطبيب يمكن أن يعمل في عدة عيادات، والعيادة يمكن أن تضم عدة أطباء.
  • PATIENT → APPOINTMENT: المريض يمكن أن يحجز عدة مواعيد.
  • DOCTOR → APPOINTMENT: الطبيب يمكنه handling عدة حجوزات.
  • CLINIC → APPOINTMENT: كل حجز يحدث داخل عيادة معينة.
  • APPOINTMENT → MEDICAL_RECORD: كل حجز قد ينتج عنه سجل طبي واحد.

4. لماذا هذا النموذج مناسب؟

  • يغطي العمليات الأساسية للنظام.
  • سهل الفهم والشرح.
  • مناسب للـ ERD في التقرير أو العرض.
  • يعكس الهيكل الأساسي لمشروع Shefaa بدون تعقيد غير ضروري.
